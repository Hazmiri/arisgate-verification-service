# Reflection 14 – Realising There Is More Than One Way

## Date
Retrospective reflection based on project structure decisions

## Model
Rolfe et al. – What? So What? Now What?

## What?

During the project, I realised that there are many ways to build the same thing. At first, this confused me. One person might put logic inside the controller. Another person might create services. Another might add repositories and mappers. Another might use a different database or folder structure.

I kept thinking: which way is the correct way? Then I started to understand that software development is not only about one correct answer. It is about choosing an approach and being able to explain why.

For ArisGate, we separated logic into services such as risk scoring, OTP, address validation and decision logic. That made the project easier to read and explain.

## So What?

This mattered because I stopped looking for magic commands and started thinking about design decisions. The easier way is not always the lazy way. Sometimes the simple way is the professional way because it reduces confusion.

I also realised that my tutor is not only checking whether the code runs. They are checking whether I understand the shape of my own project.

## Now What?

In future projects, I will ask: does this structure help another person understand the project? If the answer is yes, it is probably a good structure. I will not try to make the code look complicated just to impress someone. Clear code is stronger than clever code.
